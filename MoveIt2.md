# 🤖 2단계: MoveIt2 실습 전 필수 핵심 기초 가이드

> **⚠️ 실무 핵심 가이드라인**
> **"전처리가 모델보다 중요합니다."** 로봇 제어도 마찬가지입니다. 화려한 상위 모션 플래닝 알고리즘을 코딩하기 전에, 로봇의 뼈대(URDF)와 좌표계(TF)라는 데이터 전처리가 완벽하게 고정되어 있어야 합니다. 이 기초 설정이 흔들리면 아무리 훌륭한 플래너도 무용지물이 됩니다. MoveIt2 문제의 80%는 알고리즘 오류가 아닌 **URDF, 좌표계, IK**의 정합성 문제에서 발생합니다.

---

## 1. 로봇팔의 움직임을 이해하는 두 가지 관점

로봇팔을 제어할 때는 어떤 공간을 기준으로 명령을 내릴 것인지 명확히 구분해야 합니다.

### 1️⃣ Joint Space (관절 공간 기준)
* **개념:** 로봇팔의 각 관절 구동축에 직접 각도값(라디안)을 주어 움직이는 방식입니다.
* **초보자 비유:** 내 팔을 움직일 때 "어깨 30도, 팔꿈치 45도 굽혀"라고 직관적으로 각 관절을 조절하는 것과 같습니다.

### 2️⃣ Task Space (작업 공간 기준)
* **개념:** 로봇이 작업하는 3차원 공간 좌표를 기준으로 움직이는 방식입니다.
* **초보자 비유:** 각 관절의 각도에 상관없이 "테이블 위의 컵을 집으러 (x, y, z) 위치로 가라"고 명령하는 것과 같습니다.

> **💡 실무 핵심 포인트**
> 사람은 직관적인 **'좌표(Task Space)'**로 명령을 내리지만, 로봇 구동기는 결국 **'관절(Joint Space)'**의 모터를 회전시켜야 움직입니다. **MoveIt2는 이 두 공간 사이의 간극을 자동으로 계산하고 연결해 주는 핵심 엔진입니다.**

## 2. 정기구학(FK) vs 역기구학(IK)

두 공간 사이를 전환하기 위해 반드시 통과해야 하는 수학적 변환 알고리즘입니다.



### 1️⃣ 정기구학 (Forward Kinematics, FK)
* **정의:** 관절 각도 정보를 바탕으로 로봇 손끝(End Effector)의 위치와 자세를 계산하는 과정입니다.
* **흐름:** $\text{Joint 값} \rightarrow \text{End Effector 위치/자세}$
* **예시:** `joint1 = 30도`, `joint2 = 45도` 일 때, 최종 손끝은 특정 3차원 좌표로 명확히 도출됩니다. (해는 언제나 단 하나 존재)

### 2️⃣ 역기구학 (Inverse Kinematics, IK)
* **정의:** 우리가 원하는 목표 위치와 자세에 도달하기 위해, 각 관절이 몇 도가 되어야 하는지 역으로 계산하는 과정입니다.
* **흐름:** $\text{목표 위치/자세} \rightarrow \text{필요한 Joint 값 계산}$
* **매커니즘:** MoveIt2 내부의 **IK Solver**가 수많은 대수적/수치적 연산을 통해 관절값을 찾아내면, **Planner**가 이를 바탕으로 장애물을 피하는 최적의 경로를 생성합니다.

> **⚠️ 실무에서 자주 터지는 IK 트러블슈팅**
> * **IK 해 없음 (No IK Solution):** 로봇이 물리적으로 도달할 수 없는 작업 영역(Workspace) 밖의 위치를 명령했을 때 발생합니다.
> * **특이점 및 비정상 자세:** 수학적 해는 존재하나, 팔이 완전히 뒤집히거나 꺾이는 기괴한 자세가 채택되어 모터나 기구부에 과부하를 줍니다.
> * **다중 해 불연속성:** 동일한 좌표에 도달할 수 있는 관절 조합이 여러 개일 때, 일관성 없는 해를 선택하면 로봇이 엉뚱하고 위험한 경로로 휙휙 돌아가는 현상이 발생합니다.

## 3. URDF (로봇의 물리적 뼈대 정의)

* **한 줄 정의:** XML 형식을 사용하여 로봇의 물리적 구조와 외형을 기술한 **로봇 구조 설명서**입니다.
* **핵심 요소:** * **Link (링크):** 로봇의 단단한 뼈대를 정의합니다. (시각/충돌 모델 포함)
  * **Joint (조인트):** 뼈와 뼈를 연결하는 관절을 정의하며, 회전축 및 가동 범위를 설정합니다.

### 📄 초간단 URDF 문법 구성 예시

    <robot name="simple_arm">
      <link name="base"/>
      
      <link name="arm"/>

      <joint name="joint1" type="revolute">
        <parent link="base"/>  # 부모 링크 지정
        <child link="arm"/>    # 자식 링크 지정
        <axis xyz="0 0 1"/>    # [실무 포인트] 회전축의 방향 벡터를 지정합니다. (여기서는 Z축 회전)
      </joint>
    </robot>

> **🔥 실무자 다발생 실수**
> * **축(Axis) 방향 오류:** 시각적으로는 맞아 보이나 실제 회전축 벡터 설정이 틀려, 모터를 돌리면 로봇이 엉뚱한 평면으로 회전합니다.
> * **링크 길이 오차:** 설계 도면과 다른 수치를 입력하면 IK 계산 자체가 전부 틀어져 손끝이 목표점에 절대 도달하지 못합니다. (URDF는 시각물보다 수학 모델로서의 정확성이 훨씬 중요합니다.)
> * **충돌 모델(Collision Model) 누락:** 눈에 보이는 비주얼 데이터만 넣고 물리적 충돌 영역을 지정하지 않으면, 자가 충돌을 감지하지 못하고 자신의 몸체나 주변 벽을 그대로 뚫고 지나가 파손됩니다.

## 4. SRDF (MoveIt 전용 동작 설정 문서)

* **한 줄 정의:** 물리 구조를 다룬 URDF 위에, **"이 로봇을 MoveIt 환경에서 어떻게 제어하고 최적화할 것인가"**를 기술한 상위 설정 파일입니다.

### 📊 URDF와 SRDF의 핵심 역할 비교

| 구분 | URDF | SRDF |
| :--- | :--- | :--- |
| **핵심 역할** | 로봇의 물리적 구조, 링크 길이, 외형 정의 | MoveIt 모션 플래닝을 위한 제어 그룹 및 환경 설정 |
| **주요 기능** | `<link>`, `<joint>`, 축 방향, 충돌 범위 기술 | Planning Group 정의, End Effector 매칭, 자가 충돌 무시, 기본 자세 정의 |
| **실무 관점** | **이 파일이 없으면 로봇 자체가 존재하지 않음** | **이 파일이 없으면 MoveIt이 로봇을 제어 못 함** |

---

## 5. Planning Group 및 End Effector

MoveIt2가 복잡한 다자유도 로봇을 효율적으로 계산하도록 구획을 나누는 필수 개념입니다.

### 1️⃣ Planning Group (경로 계획 그룹)
* **정의:** 모션 플래닝을 할 때 **"동시에 계산하여 움직일 관절들의 묶음"**입니다.
* **실무 구성 전략:** 일반적으로 팔 전체를 움직이는 `arm` 그룹과 물건을 잡는 `gripper` 그룹을 철저히 분리합니다. 이 둘을 모호하게 하나로 합쳐서 플래닝하면, 손가락을 굽히는 사소한 동작 때문에 팔 전체의 기구학 연산이 다시 실행되어 경로 생성 속도가 극도로 저하되거나 궤적이 꼬이는 실수가 자주 발생합니다.

### 2️⃣ End Effector (말단 장치)
* **정의:** 로봇팔의 가장 끝단에 장착되어 실제 환경과 상호작용하는 **"로봇의 손"**입니다. (예: 그리퍼, 흡착패드, 드릴 등)
* **실무적 중요성:** MoveIt2에 목적지 좌표를 줄 때 언제나 **이 End Effector의 중심점**을 기준으로 연산이 수행되므로, SRDF 내에서 정확하게 부모 링크와 말단 장치 간의 관계가 정의되어 있어야 합니다.

## 6. 좌표계 변환 시스템 (TF, Transform)

> **🎙️ 실무 강의 핵심 멘트**
> **"MoveIt2 제어는 시작부터 끝까지 좌표계(TF) 싸움입니다."**

MoveIt2 환경 내의 모든 데이터는 상대적인 변환 관계(TF) 구조를 가집니다. 로봇 제어 시 반드시 마주하게 되는 필수 트리 구조는 다음과 같습니다.

$$\text{world (세계 기준)} \rightarrow \text{base\_link (로봇 기준)} \rightarrow \text{arm (관절)} \rightarrow \text{tool0 / end\_effector (손끝 기준)}$$

* **좌표계 오인 매칭:** 타겟 좌표의 기준축(`frame_id`)을 `world`가 아닌 로봇의 `base_link`나 카메라 좌표계로 오인하여 지정하면, 로봇이 목표와 완전히 반대 방향으로 돌진합니다.
* **회전(RPY) 이해 부족:** 단순 $(x, y, z)$ 위치뿐만 아니라 진입하는 각도인 Roll, Pitch, Yaw 제어 데이터를 명확히 매칭해 주어야 비정상 자세로 인한 플래닝 실패(실행 자체 실패)를 막을 수 있습니다.

## 7. MoveIt2 제어 프로세스 전체 아키텍처

우리가 명령을 내린 순간부터 로봇이 물리적으로 움직이기까지 내부에서 일어나는 유기적인 흐름입니다. 이 시퀀스를 머릿속에 넣어두어야 디버깅이 가능합니다.
